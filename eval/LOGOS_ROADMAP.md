# SEIF + LOGOS: One Honest, Prioritized Roadmap

## 0. The frame that disciplines everything below

The measured arc is the whole story: **blind 14/20 (70%) → Bv1 11/20 (55%) → Bv2 15/20 (75%)**. Two facts are load-bearing and must not be smoothed over:

1. **The win was the *stop signal*, not the *patch*.** Bv1 lost to blind because a narrow self-authored repro became a false oracle that terminated the loop early (false-positive green → shipped incomplete). Bv2 recovered +20pp by moving the stop decision *out of the worker's context* into an independent judge. The +20pp over v1 is real, high-confidence, a strict superset. The +5pp over blind is **a tie at n=20/1-seed (p=1.0)** — inside the noise floor.
2. **The judge is provably not an oracle.** On pytest-5840 it ruled COMPLETE while the hidden tests failed — a shared-blind-spot failure (the worker never localized to the right region; an LLM reading the same diff shares the same priors).

Everything credible follows from those two facts.

---

## 1. The single biggest lever

**Replace the LLM-opinion stop signal with execution-grounded verification — and demote every LLM judgment (single gate or panel) to a tie-breaker.**

Why this and not anything else:

- The measured bottleneck is the **stop/select decision**, not the worker's raw patching ability (v1→v2 proved fixing the signal alone bought +20pp).
- Bv2's gate is the *strongest version of the wrong category*: it is one LLM's text opinion, correlated with the worker's blind spots, and you have a measured instance (pytest-5840) where it fails exactly where it matters.
- The one signal you have *not* yet used is **real code execution beyond the agent's own repro** — the repo's own adjacent tests. That signal is (a) uncorrelated with the LLM judge, (b) grounded in ground-truth behavior, and (c) attacks the precise false-COMPLETE class the gate cannot see.

This is also why the **verifier panel (Codex + Gemini) is the most overrated move on the table** and is *not* the lever: a second LLM helps only when judges disagree, but your real failures are cases where both judges agree and both are wrong. Paying 2× for correlated opinions (and Gemini is 403 anyway) buys the *illusion* of independence. Add the execution signal first; only consider a panel after you have *measured* judge-disagreement rate and found it non-trivial.

## 2. Phased plan

### NEAR-TERM (Phase 1) — Make the v2 result believable, then add the one uncorrelated signal

This phase contains **no resolve-rate claim**. Its job is to convert "direction + pattern" into a variance-bounded estimate, and to add the single highest-EV new signal.

**Build 1a — Statistical floor: 3 seeds × the existing 20, plus the cheap claim-hardening ablations.**
- Wire `seed`/`temperature` (or, if `claude -p` exposes neither, treat each independent run as a seed and record run-index — honest either way). Re-run A / Bv1 / Bv2 across 3 runs. Extend `analyze.py` (already does Wilson + exact McNemar) to report per-seed rate, mean ± SD, pooled-seed McNemar.
- Add the **multi-turn-no-gate ablation**: identical to Bv2 but with the independent gate removed (loop self-critiques to budget). This is the one experiment that decides whether you have a moat: if no-gate ≈ Bv2, your edge is "more compute," not "independent verification." If gate > no-gate, you have isolated evidence that *independent verification specifically* works — the System-2 thesis, earned.
- Add a **contamination/cutoff split**: base_commit date vs model knowledge cutoff, report resolve rate pre- vs post-cutoff. Single-file requests/pylint fixes are the most memorization-suspect; a high number there may be retrieval, not reasoning.
- *Expected effect:* zero resolve-rate lift by design; tighter CIs, and answers to "is the +5pp real?" and "is the gate the moat or just tokens?"
- *Honest risk:* the likely uncomfortable result is that Bv2 ≈ blind within noise on this subset — i.e., the gate *recovers v1's regression* but does **not** yet beat blind. That is still true and publishable, and better known now than after building on a phantom edge.

**Build 1b — Regression-test feedback: run the repo's own adjacent test modules, feed failures back.**
- Select tests by import/path proximity to changed source; run before/after each candidate patch in the locked sandbox; a previously-passing test that now fails = hard reject; under-coverage feedback drives the next turn.
- Pair with the cheap **differential gate**: any agent-authored test must FAIL pre-patch and PASS post-patch (kills the v1 "narrow repro that passes on a partial fix" pathology directly). Skip the expensive mutation-score half for now — slow, flaky, and the test-author is incentivized to write weak tests; revisit as a stretch experiment, not a gate.
- **Non-negotiable reward-hacking guard (this is the single most dangerous item in the whole plan):** the related-test set must be a **hard, audited set-difference** against each instance's graded FAIL_TO_PASS / PASS_TO_PASS ids, asserted in code (extend `swe_common`'s test guard) as a **build-breaking error**, with `selected ∩ graded = ∅` logged to the ledger per instance. If proximity selection ever pulls a graded test, you have leaked the oracle and every downstream number — including ones already reported — is void.
- *Expected effect:* the first plausible *resolve-rate* gain, concentrated in the multi-file bucket where you're losing. It catches the complementary failure to v1 (patches that fix the target but silently regress PASS_TO_PASS, which nothing currently gates on).
- *Honest risk:* near-silent when the repo's existing tests are thin around the changed code (often true — that's why the bug existed); flaky/slow tests add cost. It is a *precision* boost (catches breakage/under-coverage), not a recall boost (won't conjure an approach the worker never tried).

### MID-TERM (Phase 2) — Attack the *generation* bottleneck and rebuild the eval where the thesis lives

Phase 1 closes the false-positive-stop hole; the residual losses are **coverage** failures — the gold approach was never in a pool of one. No judge selects a correct patch from a pool of one wrong patch.

**Build 2a — Best-of-N with diverse generators and an execution-grounded ranker.**
- Generate N=3–5 candidates with *real* diversity: different model families (Claude + Codex-as-author; Gemini when live) and varied localization prompts ("fix the narrowest cause" vs "find every code path the issue touches") — **not** N temperature samples of one model, or pass@N barely exceeds pass@1.
- Rank by a deterministic, execution-led key: `PASS_TO_PASS regression survival > differential/repro-pass > multi-path coverage when the issue is multi-path > LLM gate verdict (tie-break only) > diff parsimony`. The LLM never primary-selects.
- **Report pass@1 (the only legitimate capability claim) and pass@N (oracle upper bound — labels as such) separately**, plus **cost-per-resolved-instance**. Run the equal-budget control (diverse-N vs same-model-N) to prove diversity beats more-of-the-same.
- *Expected effect:* the highest-ceiling resolve-rate move, because it attacks generation, which the judge-only path cannot.
- *Honest risk:* worst cost-per-point (3–5× × seeds × regression runs — a real BYOK bill; sequence it last). A weak ranker actively *hurts* pass@1 at higher N — which is exactly why Phase 1's execution-grounded ranker is a hard prerequisite. If candidate edit-regions overlap heavily (correlated generators), pass@N ≈ pass@1 and you've burned compute — measure region overlap and stop if so.

**Build 2b — Rebuild the eval set where the thesis actually lives.**
- The current 20 is single-file-heavy and the load-bearing *large* bucket has **n=2** — you cannot measure a degradation slope on n=2. Pull multi/large instances from heavier repos (django, sympy, matplotlib, sphinx). Report resolve rate **per horizon bucket** (files touched / hunks / FAIL_TO_PASS count), and the **slope** of resolve-vs-horizon as the headline, with CIs.
- *Expected effect:* makes the real (long-horizon) number legible. Will likely *lower* the raw number — that is the point; pre-commit to reporting it.
- *Honest risk:* if the slope is already flat on SWE-bench, that is a *finding*, not a win — it means SWE-bench is too short to test the context-collapse thesis and you need a genuinely longer-horizon eval. Do **not** claim a "flat-curve win" from a benchmark too short to induce collapse.

**Build 2c — Calibrate the gate against ground truth (continuous, unlocked by seeds).**
- Log every signal's verdict + the harness ground-truth result; compute per-signal and per-combination FP rate (says COMPLETE, harness FAILs) and FN rate (withholds on a correct patch). Produce the gate's confusion matrix / ROC; pick the COMPLETE threshold from data (bias toward *continue-on-doubt* — a false-INCOMPLETE costs iterations, a false-COMPLETE ships a broken patch).
- *Expected effect:* converts "I added a gate" into "my gate has a measured X% FP rate at this threshold" — the thing that lets you tune honestly.
- *Honest risk:* meaningless until n is large enough; don't overfit a threshold to 20 instances; light-repo calibration won't transfer perfectly to heavy repos.

### LONGER-TERM (Phase 3) — Long-horizon machinery, gated on a measurable slope

This is the founder's actual goal (no context collapse on long-horizon tasks) — and it is currently **unmeasured and possibly unmeasurable on SWE-bench**. Build it **only after Phase 2b shows a real degradation slope worth flattening.**

**Build 3a — SEIF ledger as the worker's only durable state; each HTN subgoal runs in a fresh, bounded context.**
- Each subgoal = a fresh `claude -p` whose input is reconstructed from the ledger (typed envelope + compact world-state projection: files touched, tests now passing, prior verdicts, open obligations) — not the accumulated transcript. Per-step context becomes O(1) in task length instead of O(length); that is the mechanism that turns decay into a flat line.
- *Honest risk:* highest upside *and* highest risk. On short tasks fresh-context decomposition can **lose** to a single long context (handoff seams). It will likely *lower* the light-repo number; do not default it until the slope chart shows a crossover where it wins.

**Build 3b — ATMS-style rollback + per-subgoal checkpoints.**
- Every committed subgoal is a restorable checkpoint (git worktree + ledger hash). On verification failure, rewind to the last good checkpoint and try an alternate decomposition (beam=2, depth-limited) rather than restarting. This is the term that *recovers* error on the curve, so length stops being purely multiplicative.
- *Honest risk:* combinatorial blow-up — cap branches/iterations and report cost-per-solve. Only pays off if failure detection is good, so it *depends on* Phase 1's execution signals firing.

**Explicitly parked (research bets, not roadmap):** HTN subgoal *contracts* with machine-checkable postconditions (real issues don't decompose into contracts before you've read the code; risk of FSM deadlock and *lower* resolve rate); MCTS/tree search over patches (needs a cheap, trustworthy value function — you have the opposite); property/metamorphic testing as a core gate; self-consistency *as a stop signal* (correlated agreement = false confidence — build it only as a diminishing-returns diagnostic). Verifier *panels* stay parked until measured judge-disagreement justifies them.

## 4. The honest bottom line

- **Biggest lever:** execution-grounded verification (regression + differential signal), LLM opinions demoted to tie-breakers — it attacks the one *measured* hole (pytest-5840 false-COMPLETE) and generalizes the one *measured* win (externalizing the stop decision bought +20pp).
- **Sequencing:** Phase 1 (seeds + no-gate ablation + regression/differential signal, with the hard reward-hacking guard) → Phase 2 (diverse best-of-N with an execution-grounded ranker + heavier-repo eval + gate calibration) → Phase 3 (fresh-context HTN + rollback, only once a slope is measurable).
- **Stop doing now:** reporting/iterating at n=20/1-seed; polishing the LLM gate as if it were the answer; planning LLM verifier panels before measuring judge correlation; treating the single-file bucket as evidence of reasoning; chasing the 500-instance leaderboard on a scaffold that changed twice this session.
- **Honest ceiling:** a credible **high-70s to low-80s MEASURED** on these light repos, with the real gains in the multi-file bucket and at meaningful cost. **~100% is not reachable by inference-time search/verification alone** — the withheld-test discipline that makes your number *honest* is the same thing that caps it below an oracle, and a real fraction of failures are generation-bound (out of the base model's reach at any N). The durable, uniquely-yours, not-already-saturated headline is **a measurably flatter degradation slope backed by an execution-grounded, never-self-reported verification stack with a published FP/FN rate, CIs, and $/solve** — that is the "own + prove your work" moat, and it survives scrutiny in a way a single resolve-% never will.

# Stress-Test: The Combined SEIF + LOGOS Plan

The five proposals converge on a near-identical move set under different banners. Deduplicated, the combined plan is really **six things**: (1) seeds/multi-run statistical power, (2) regression-test feedback (run the repo's own adjacent tests), (3) best-of-N with diverse generators + a selector, (4) verifier panels / ensembles, (5) gate calibration / ablation harness, (6) long-horizon machinery (fresh-context HTN + ledger state + rollback + contracts + heavier repos). Below I judge each against the only thing that matters: does it move *harness-scored, held-out* resolve rate, or does it just move the *story*.

---

## (a) Moves most likely to be hype / won't move the needle

**1. Verifier panels (Codex + Gemini ensemble of LLM judges) — the most overrated move in the entire plan.** Every proposal that pushes this also quietly admits the killer: your one measured gate failure (pytest-5840) was a **shared-blind-spot** failure — the *worker* never localized to `config/__init__.py`, and an LLM judge reading the same diff shares the same training-distribution priors about where the bug lives. Adding a second LLM judge only helps when judges *disagree*; on the failures you actually have, they will *agree and both be wrong*. Worse, the math sold for it ("joint FP → p₁·p₂") assumes independence you do not have and have not measured. Correlated LLM judges multiply cost, not coverage. Proposal 1 is the only one honest enough to demote the LLM gate to a tie-breaker; everyone else over-invests. **Verdict: low-leverage; do not build a panel until you've measured judge-disagreement rate and found it non-trivial. Right now you'd be paying 2× for the illusion of independence — and Gemini is 403 anyway, so half of it is vapor.**

**2. Self-consistency / patch-cluster agreement (Proposal 1, Move 4).** Same disease, dressed as confidence. Agreement among same-family samples is *correlated* agreement. On genuinely ambiguous issues the model will confidently converge on a plausible-wrong fix — the exact dual of the pytest-5840 false-COMPLETE. It's a nice *instrument* for measuring diminishing returns, but as a *gate* it manufactures false confidence. Build it as a diagnostic, never as a stop signal.

**3. HTN subgoal contracts with machine-checkable postconditions (Proposal 3, Move 5).** This is the most speculative item and its own author sequences it last and flags it will likely *lower* resolve rate. Real SWE-bench issues don't decompose into contract-able subgoals before you've read the code — you discover the structure by reading. Premature contracts force a bad decomposition and can deadlock the FSM. **Verdict: research bet, not a roadmap item. Park it.**

**4. Property-based / metamorphic testing as a core gate.** Proposals 2 and 3 both raise it and both immediately retract it. Correct instinct. Most SWE-bench bugs are specific behavioral defects without clean invariants; authoring good properties is high-variance hand-work. Opportunistic booster on parsers/serializers at best. Not a workhorse.

**5. Mutation testing scoped to the diff (Proposal 2, Move 3) — partial hype.** The *differential* half (new test must fail pre-patch, pass post-patch) is genuinely cheap and good — keep it. The *mutation-score* half is where it inflates: mutation runs are slow and flaky even scoped to hunks, and the agent that authors the tests is incentivized to write tests that survive its own weak fix. It's a real signal but the cost/robustness ratio is poor on a single-founder rig. **Keep differential; treat mutation as a stretch experiment, not a gate.**

## (b) The honest performance ceiling for a sovereign single-founder setup

**Realistic MEASURED ceiling on these light repos: high-70s to low-80s%.** Multiple proposals land here independently and the convergence is meaningful. Over a credibly-built version of this plan, expect to claw a few points over v2's 75%, mostly in the multi-file bucket, *and at n=20/1-seed it will read as noise.*

**~100% is not realistic, and the reason is structural, not effort-related:**

1. **Your verifier is provably not an oracle, by design.** You correctly withhold the held-out tests (reward-hacking defense). That means at solve time you are *always* selecting against a proxy. There is an irreducible class of patches that are plausible, touch the right region, pass your repro and your adjacent regression tests, and are still wrong on a hidden edge case. No amount of N, no panel, no tree search recovers that class without seeing the oracle — and seeing the oracle voids the eval. **The withheld-test discipline that makes your number honest is the same thing that caps it below 100%.** This is a load-bearing tension every proposal gestures at but none states this bluntly.

2. **A meaningful fraction of SWE-bench-Verified failures are generation-bound, not selection-bound.** When the gold approach is outside the base model's reach at any temperature, best-of-N raises pass@N but pass@1 plateaus — the gap is the selector's ceiling, and the selector is your imperfect gate. Inference-time search cannot manufacture an approach the model never proposes. Moving that floor requires a stronger base model or training, which is off the table for a sovereign BYOK setup.

3. **Frontier published numbers (high-70s–80s+ on full Verified) come with massive scaffolds, model access, and engineering you cannot match solo.** A self-hosted single-founder rig landing a *defensible, CI-bounded* high-70s on a fixed subset is a genuinely respectable result. Claiming ~100% would be the exact overclaim your doctrine exists to prevent.

**The honest headline is not a resolve-% at all — it's the degradation slope** (does capability stay flat as task length grows). That's the only number that is both uniquely yours and not already saturated by frontier agents. But see risk (e) — it is currently *unmeasured*, and may be unmeasurable on SWE-bench.

**1. Reward hacking via the regression-test selector — the single most dangerous item in the plan.** Multiple proposals propose selecting "adjacent" repo tests by import/path proximity. **If your proximity heuristic ever pulls a graded FAIL_TO_PASS or PASS_TO_PASS test into the feedback loop, you have leaked the oracle and every downstream number is void — including the ones you've already reported.** This is not hypothetical; it's the most likely way this specific plan silently corrupts itself. Mitigation must be a *hard, audited* set-difference against each instance's graded test IDs, enforced in code (extend `swe_common`'s test guard), with a per-instance assertion that selected ∩ graded = ∅, logged to the ledger. **2. Contamination / memorization, especially on the easy bucket.** Only Proposal 5 takes this seriously, and it's right. **3. The verifier-is-not-an-oracle problem is treated as solvable; it isn't.** The plan's center of gravity is "make a better proxy oracle." Fine — but several proposals drift toward language implying the gate can be *trusted*. It can't, ever. The only correct framing is "multiple imperfect, ideally *uncorrelated* signals, with a measured FP/FN rate." Execution signals (repro, regression) are uncorrelated with the LLM judge and therefore worth more than another LLM vote — that's the real lesson, and it argues *against* the panel and *for* execution grounding.

**4. Compute cost is under-priced across the board.** Best-of-N (3–5×) × diverse generators × seeds (3×) × verifier panel × per-instance mutation × heavier repos is a multiplicative blowup on a `claude -p` BYOK budget. None of the proposals put a dollar figure on the full stack. **You must report cost-per-resolved-instance alongside resolve rate** — a +5pp at 5× compute is a different (worse) product, and on a solo rig it may simply be unaffordable to run at the n needed for significance.

**5. Context-collapse is NOT solved by this plan — and may not even be measurable on SWE-bench.** This is the most important strategic risk and only Proposal 3 confronts it honestly. The entire long-horizon thesis (fresh-context HTN + ledger-as-state + rollback) is **unvalidated**, and the load-bearing bucket (large, multi-file) has **n=2**. Worse: SWE-bench-Verified instances are hours-scale, not days-scale — the slope may be *flat already* simply because the benchmark is too short to induce collapse. If so, a "flat curve win" would be an artifact, not evidence. **The plan risks building elaborate anti-collapse machinery (fresh-context decomposition, contracts, ATMS rollback) that demonstrably *lowers* resolve rate on short tasks while never being validated on the long tasks it's for — because you don't have a long-horizon benchmark.** That is the biggest hidden hole: *the moat you're selling has no measurement instrument.*

## The 3 moves that survive scrutiny as genuinely high-leverage

**Move 1 — Statistical floor first: 3 seeds × the existing 20, plus the cheap claim-hardening ablations (multi-turn-no-gate, contamination/cutoff split).**
This raises *credibility*, not resolve rate — and credibility is half the stated goal ("every claim provable"). It's the precondition for believing anything else. Critically, it includes the **no-gate ablation**, which answers the one question that decides whether you have a moat or not: *is v2's edge the independent verifier (the System-2 thesis) or just more compute?* If no-gate ≈ v2, your selling point collapses to "spend more tokens" and you need to know that before building anything on top of the gate. Low effort, highest information-per-dollar. Do this before writing another feature.

**Move 2 — Regression-test feedback: run the repo's own adjacent tests (audited non-overlap with graded IDs), feed failures back.**
This is the only proposed move that adds a signal **uncorrelated with the LLM judge and grounded in real execution** — a too-narrow patch that breaks neighboring behavior fails *code*, not a reviewer's opinion. It directly attacks the false-COMPLETE class the gate provably can't see, and its expected lift concentrates exactly where you're losing (multi-file). It is also the move that *most demands* the hard reward-hacking guard from risk (c.1) — build the audited set-difference into the same change, as a build-breaking assertion. Pair with the cheap *differential* test gate (new test must fail pre-patch, pass post-patch); skip the expensive mutation half for now.

**Move 3 — Best-of-N with DIVERSE generators and an execution-grounded ranker (regression-pass > repro-pass > LLM verdict as tie-breaker only).**
This is the only move that attacks the *generation* bottleneck, which is where your residual losses actually live (the gold approach was never in a pool of one). The non-negotiable design constraints, distilled from the stress-test: (i) diversity must be *real* — different model families / localization prompts, not N temperature samples of one model, or pass@N barely exceeds pass@1; (ii) the ranker must lead with **execution signal** (Move 2's regression result), with the LLM gate demoted to tie-breaker — a weak LLM ranker actively *hurts* pass@1 at higher N; (iii) **report pass@1 (the only legitimate capability claim) and pass@N (oracle upper bound) separately**, and cost-per-resolved-instance, or you'll fool yourself. Highest ceiling, worst cost — do it last, after Move 2 makes the ranker execution-grounded.

**Sequencing:** Move 1 (now, cheap, gates everything) → Move 2 (the real uncorrelated-signal lift) → Move 3 (highest ceiling, only worth it once the ranker is execution-grounded).

**The one-line bottom line:** the durable, honest win is *not* a higher resolve-% — it's an **execution-grounded, never-self-reported verification stack with a measured FP/FN rate, reported with CIs and cost**. Everything that adds *another correlated LLM opinion* (panels, self-consistency gates, contracts) is motion, not progress. And the long-horizon "no context collapse" moat — the actual goal — remains **unmeasured and possibly unmeasurable on SWE-bench**; until you have a benchmark that exhibits a degradation slope, do not build the machinery for it, and do not claim it.
